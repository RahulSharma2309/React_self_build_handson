# 10 — Checkout Forms Questions

Use these after completing Story 10.

## Questions

1. What is a controlled input?
2. Why should form values live in React state?
3. What does `onChange` do in a controlled field?
4. What does the `value` prop do in an input?
5. What bug appears if an input has `value` but no working `onChange`?
6. Why should checkout read cart items from Redux?
7. What should happen if the user opens checkout with an empty cart?
8. What fields should the customer form collect?
9. Where should validation errors be shown?
10. Why should validation not exist only in `console.log`?
11. What is form submit state?
12. Why should you prevent the default browser form submission?
13. What is a ref?
14. Why is focusing the first invalid field a good ref use case?
15. Why should refs not replace normal form state?
16. What should the order object contain before sending it to an API?
17. What data comes from the form, and what data comes from the cart?
18. How would you debug an input that does not update while typing?
19. How would you debug a submit handler that never runs?
20. In an interview, how would you explain controlled vs uncontrolled inputs?
21. Why is accessibility important in forms?
22. Why should labels be connected to inputs?
23. How should error messages help keyboard and screen reader users?
24. What should happen after a successful checkout in the next story?
25. How does this story connect UI state, global state, and user actions?
26. Why should `event.preventDefault()` be used on form submit?
27. What form data is local state?
28. What checkout data comes from Redux?
29. What checkout values are derived?
30. Why should validation include cart contents?
31. Why should refs not store input values?
32. What is a good ref use case in checkout?
33. What should the order object contain?
34. Why should checkout not clear the cart before backend success?
35. How would you explain checkout state flow in an interview?
